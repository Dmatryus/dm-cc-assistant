# v0.3.0 — Design Notes

Working document. Captures architectural decisions made during design discussion.
Subject to refinement until v0.3.0 is implemented and released.

---

## 1. Концепция

«Долго думаем → автономно делаем → структурированный отчёт → диалог».

Задачи становятся крупнее (эпики), агент сам запускает параллельные имплементации в worktree'ах. Точки взаимодействия с пользователем — до старта (планирование) и после (отчёт).

---

## 2. Внешний интерфейс — 4 команды

| Команда | Что делает | Интерактивность |
|---|---|---|
| `/backlog` | Генерирует / показывает эпики, sync с docs | Лёгкий диалог |
| `/plan E-001` | Декомпозиция эпика на подзадачи, DAG, волны, риски, DoD | Тяжёлый интерактив |
| `/execute E-001` | Параллельный прогон подзадач + диалог по отчёту | Автономно + диалог в конце |
| `/release E-001` | `feat/E-001` → main, CHANGELOG, version bump, tag | Подтверждения |

Старые `/research`, `/review`, `/update-docs` снаружи **исчезают**. Соответствующие skills удаляются.

## 3. Агенты

| Агент | Статус в v0.3 | Роль |
|---|---|---|
| `backlog-planner` | Обновить | Генерит эпики, sync, suggestions, миграция |

## 4. Backlog — модель данных

### ID-схема
- Эпик: `E-001`
- Подзадача: `E-001.1` (точечная нотация — родитель виден из ID)

### Статусы

| Уровень | Значения | Где хранится |
|---|---|---|
| Эпик | `Todo` / `In Progress` / `Done` | Через секцию в `backlog.md` |
| Подзадача | `Todo` / `In Progress` / `Done` / `Failed` / `Blocked` | Inline в строке подзадачи |

`Failed` и `Blocked` — для cascade-кейса при автономном выполнении.

### Тип эпика (опциональный inline-лейбл)
- `Functional` (default, можно опустить)
- `Tech Debt`
- `Research`
- `Infrastructure`
- `Cleanup`

### Параллелизм
- Внутри эпика: волны подзадач (параллельные worktree'ы)
- Между эпиками: **один эпик In Progress за раз** (линейный pipeline)

### Размер эпика
**Variant C (закреплён):** эпик = логическая группа задач с самостоятельной ценностью. Размер плавающий, определяет пользователь. Нет жёсткой привязки к minor-релизу.

## 5. Артефакты

| Файл | Создаётся / обновляется | Содержит |
|---|---|---|
| `.task/backlog.md` | `/backlog` + все шаги | Эпики + подзадачи + статусы + parking lot + rejected |
| `.task/plan-E-001.md` | `/plan` | Контекст, риски, альтернативы, DoD, DAG, open questions |
| `.task/report-E-001.md` | `/execute` (агрегат от execution-agent'ов) | Done / Decisions / Failed / Skipped / Open questions / Follow-ups |

---

## 6. Модель веток

```
main
 └─ feat/E-001                      (создаёт /execute)
     ├─ feat/E-001.1 worktree       (волна 1)
     ├─ feat/E-001.2 worktree       (волна 1)
     └─ feat/E-001.3 worktree       (волна 2, после merge волны 1)
```

### Жизненный цикл веток
1. `/execute` создаёт `feat/E-001` от main.
2. Execution-agent владеет своим worktree.
3. Между волнами `release-manager` мёрджит подзадачи в `feat/E-001`.
4. Следующая волна стартует с обновлённого `feat/E-001`.
5. `/release E-001` финально мёрджит в main.

---

## 7. Поведение при преграде (cascade по DAG)

1. Подзадача упёрлась в непреодолимое → останавливается, пишет в свой отчёт.
2. Параллельные в текущей волне — добегают.
3. В следующих волнах: зависимые от упавшей → `Skipped`, независимые → запускаются.
4. `/execute` финиширует в `partial`, диалог с пользователем по агрегированному отчёту.

## 8. `/backlog` — детальный дизайн

### Источники для генерации эпиков

| Источник | Тип эпика | Дефолт. приоритет |
|---|---|---|
| OVERVIEW §6 Must | Functional | Critical / High |
| OVERVIEW §6 Should | Functional | Medium |
| OVERVIEW §6 Could | Functional | Low |
| OVERVIEW §9 Open Questions (блокирующие Must) | Research | Critical / High |
| ARCHITECTURE §9 Tech Debt | Technical | По `[Priority]` тегу |
| ARCHITECTURE §10 Code Hotspots | Technical | По `[Priority]` тегу |
| ARCHITECTURE §8 Constraints (требующие setup) | Infrastructure | High |
| Gap «доки vs реальный код» | Cleanup / migration | Medium |
| OVERVIEW §6 Won't + §7 Non-goals | — | Игнорируются явно |

### Объём draft
5–12 эпиков. Если получается больше — слишком мелкая нарезка, агент предупреждает.

### Сценарии запуска

#### Первый запуск (пустой backlog)
1. Read OVERVIEW + ARCHITECTURE + light code scan.
2. Если хоть одного из docs нет — **отказ**, отправка к `/dm-cc-assistant:project-init`. Никаких degraded mode.
3. Generate draft эпиков (строго из docs/code-gap, ничего «от себя»).
4. Discuss draft с пользователем (split / merge / rename / reprioritize).
5. **Suggestions phase** (опционально) — кандидаты с обоснованием.
6. Записать `.task/backlog.md`.

#### Повторный запуск (backlog есть)
1. Read existing backlog.md.
2. **Sync check:** lightweight diff между «что было бы в backlog из текущих docs» и фактическим состоянием.
3. Surface deltas → пользователь решает по каждому (см. ниже).
4. Stand. меню: `advance` / `pick from Todo` / **`add new epic`** / `quit`.

#### Миграция с v0.2 (плоский T-ID формат)
- Detect плоский формат → спрос: **migrate** (default) / **wipe** (с бэкапом в `.task/backlog.v02.md`) / **keep-legacy** (старые в секции `## Legacy`, новые эпики в Todo).

### Suggestions — что от себя предлагает агент

**Запрет:** автономно добавлять в draft.
**Разрешено:** после draft'а — отдельная фаза с обоснованием.

**Фильтр:** только сильное обоснование (видимый признак риска), не «best practice вообще».

**Три состояния:**

| Действие | Куда попадает | Будет ли предложено снова |
|---|---|---|
| Принять | Эпик в `## Todo` | — |
| Отложить | `## Parking lot` секция | Нет, лежит на виду |
| Отклонить | `## Rejected suggestions` | Нет, пока пользователь не удалит строку |

**Две линии защиты от спама:**
1. OVERVIEW §7 Non-goals (явный proactive фильтр)
2. `## Rejected suggestions` (накопительный reactive фильтр)

**Без жёсткого capa** — оставляем только фильтр на качество обоснования.

### Add new epic — ручное добавление

Запускается из меню повторного запуска `/backlog`. Не отдельный command.

1. Пользователь даёт намерение (title + 1–2 фразы).
2. **Агент читает OVERVIEW + ARCHITECTURE + текущий backlog** (light scan).
3. **Validate step** — проверка на 4 признака сомнения:
   - Конфликт с Won't / Non-goals
   - Дубликат / overlap с существующим эпиком
   - Нет источника в docs (можно, но агент уточняет)
   - Подразумеваемая зависимость от существующего эпика
4. **Если сомнений нет** → fast path с короткими уточняющими вопросами.
5. **Если есть** → диалог по каждому concern, пользователь решает.
6. Не блокирующий: «я в курсе, добавляй» проходит без дальнейших вопросов.
7. Запись в выбранную секцию (`Todo` дефолт, `Parking lot` при желании).

### Sync check — детект изменений в docs

Запускается автоматически в начале `/backlog` (когда backlog уже есть). Lightweight, не блокирующий.

**Типы deltas:**

| Что произошло в docs | Что предлагает агент |
|---|---|
| Новый Must / Should / Could / Tech Debt / Hotspot | Добавить новый эпик |
| Удалён пункт, на который ссылался эпик | Deprecate / keep / Parking |
| Изменился `[Priority]` ([Medium] → [High]) | Обновить приоритет эпика |
| Изменена формулировка / контекст | Обновить контекст эпика? (опционально) |

**Sync НЕ:**
- Не трогает In Progress / Done эпики автоматически
- Не пересоздаёт с нуля
- Не удаляет parking / rejected
- Если deltas нет — молча идёт в меню («docs and backlog in sync»)

**Edge case:** если deltas очень много (OVERVIEW переписан с нуля) → один большой выбор: sync поштучно или wipe + regenerate с бэкапом.

### Convention update: ARCHITECTURE.md §9 §10

Чтобы убрать NLP-эвристики на «блокирующий vs опциональный» Tech Debt — формат секций §9/§10 явно требует `[Priority]` тег:

```markdown
## 9. Tech Debt

- [High] <описание> — <почему блокер>
- [Medium] <описание> — <обоснование>
- [Low] <описание> — <почему не срочно>

## 10. Code Hotspots

- [High] <файл/модуль> — <чем проблемен>
- [Medium] ...
```

Уровни: **High / Medium / Low** (Critical опускаем — это уже P0, не Tech Debt).

**Правило промоушена в backlog:**

| Тег | Куда в backlog draft |
|---|---|
| `[High]` | `## Todo` как Tech Debt / Hotspot эпик |
| `[Medium]` | `## Todo` с пометкой «обсудить приоритет» |
| `[Low]` | `## Parking lot` |

**Затрагиваемые файлы (внутри будущего E-001 или отдельным эпиком):**

| Файл | Что меняется |
|---|---|
| `agents/architecture-interviewer.md` | Шаблон §9/§10 требует `[Priority]` префикс |
| `agents/docs-updater.md` | При добавлении Tech Debt / Hotspot пишет с `[Priority]` |
| `agents/backlog-planner.md` | Читает тег вместо текстовой эвристики (fallback на Medium для legacy записей) |

## 9. `/plan E-001` — детальный дизайн

### Цель

Тяжёлое интерактивное планирование одного эпика. На входе — эпик из `.task/backlog.md`. На выходе — `plan-E-001.md` (детальный артефакт) + обновлённый `.task/backlog.md` (подзадачи под эпиком, эпик `→ In Progress`).

### Что такое подзадача

**Атомарная единица работы:**
1. Самостоятельна — execution-agent доделает без обращения к пользователю.
2. Мёрджится независимо — cascade-friendly: соседняя падает, эта остаётся валидной.
3. Чёткий DoD — известно, когда «сделано».
4. Один worktree → одна ветка `feat/E-001.X`.

**Не временной критерий** (не «1 час»), а **структурный** — атомарность, merge-independence.

### Типы подзадач

`impl` / `refactor` / `test` / `docs` / `research`.

**Тесты — отдельная подзадача / финальная волна / отдельный эпик** в зависимости от характера работы. **Не парные к каждой impl-подзадаче.** Изоляция от bias обеспечивается тем, что test-агент и impl-агент — разные.

| Кейс | Где тесты |
|---|---|
| Эпик вводит новую функциональность | Финальная волна — test-подзадача (одна или несколько) |
| Эпик — чистый рефакторинг | Существующие тесты, отдельная test-подзадача не нужна |
| Эпик — Tech Debt / Cleanup | На усмотрение `/plan` |
| Нет тестовой инфраструктуры | Отдельный эпик «Test infrastructure» |

### Фазы интерактивного диалога

| # | Фаза | Стиль |
|---|---|---|
| 1 | **Context** — read docs + light code scan, summary «что я понял» | Подтверждение |
| 2 | **Decomposition** — draft подзадач 5–15 шт. с типами и файлами | Bulk: split/merge/rename/reorder |
| 3 | **DAG + waves** — Mermaid граф, группировка в волны | Корректировки |
| 4 | **Risks + alternatives** — для развилок: альтернативы, +/−, рекомендация | One-question-at-a-time |
| 5 | **Open questions** — resolve / defer / promote to research subtask | По одному |
| 6 | **DoD** — acceptance-чеклист эпика | Bulk + правки |
| 7 | **Final review** — полный план, decision point | Accept / Iterate phase X / Abort |

**Если эпик мелкий** — фазы 4–5 пропускаются автоматически.
**Если эпик слишком большой** (>15 подзадач) — агент предупреждает, рекомендует разбить.

### Глубина light code analysis (фаза 1)

| Слой | Что | Глубина |
|---|---|---|
| 1. Docs | OVERVIEW, ARCHITECTURE, CLAUDE, backlog | Полностью |
| 2. Project skeleton | `ls -R`, `git log --oneline -20`, `.gitignore` | Полностью |
| 3. Likely affected modules | По keywords из title + «Источник» эпика + упоминаниям в docs | Полностью |
| 4. Adjacent modules | Соседние / зависимые | Заголовки (имена функций/классов через grep) |
| 5. Остальное | — | Не читает |

**Транспарентность:** в конце фазы 1 показывает «Я сфокусировался на [files]. Добавить?» — у пользователя один шанс расширить read-list **до** декомпозиции.

**Адаптивность:** если в фазе 2 нужно прочитать ещё файл — читает на лету.

### Поведение в фазе 4 при «выбери сам»

| Уверенность агента | Поведение |
|---|---|
| Сильная рекомендация | Принимает сам, помечает в плане **«delegated to agent»**, кратко обосновывает |
| Слабая рекомендация | Push back один раз: «развилка влияет на [X], мой lean — Y из-за [причина]. Точно делегируешь?» |
| 50/50 | Push back более настойчиво. Если всё равно «решай сам» → берёт lean, помечает **«weak choice — re-evaluate before /execute»** |

Все agent-делегированные решения попадают в `plan-E-001.md` § «Альтернативы» с явной пометкой и пунктом **Re-visit if:**.

### Цикл итераций

**Phase 7 — единственная decision point** для кросс-фазных переходов:

| Действие | Что происходит |
|---|---|
| Accept | Файл пишется, backlog обновляется, эпик → In Progress |
| Iterate phase X | Прыгает в X, переоткрывает, **re-flows** downstream фазы (показывает дельты, не весь поток) |
| Abort | Ничего не пишется, эпик остаётся в Todo. Контекст диалога не сохраняется. |

Внутри фазы — свободная навигация (split/merge подзадач). Между фазами — только через phase 7.

**Soft cap:** третья итерация той же фазы → агент сигналит «возможно эпик слишком большой / нечётко определён», предлагает разбить или доуточнить docs.

### Research subtask — контракт

**Когда:**
- Открытый вопрос, требующий исследования перед impl
- Развилка без явной рекомендации (фаза 4 не зарезолвила)
- Spike / выбор библиотеки / прототип

**DoD:** файл `.task/research-E-001.X.md` с фиксированной структурой:

```markdown
# Research E-001.X: <вопрос>

## Question
## Options (с +/−)
## Decision
## Rationale
## Implications (что это значит для зависимых impl-подзадач)
```

**Worktree и ветка:** как у impl. В коммите — research-файл (+ опционально код прототипа).

**Интеграция:** мёрджится в `feat/E-001`, downstream impl-подзадачи читают из integration ветки.

**Failure:** если research-агент не пришёл к decision → подзадача `Failed`, cascade на зависимых.

**Bias:** тот же агент потом не делает impl своей research'нной фичи (research-агент → impl-агент в следующей волне читает решение и делает).

### Формат `.task/plan-E-001.md`

```markdown
# Plan — E-001 [High] (Functional) <Title>

Дата: YYYY-MM-DD · Статус: Final

## 1. Контекст

## 2. Подзадачи

### E-001.X [Priority] (type) <Title>
- Что делает: ...
- Файлы: ...
- DoD: ...
- Ветка · Волна · Зависит от

## 3. DAG (Mermaid)

## 4. Волны

## 5. Риски (с mitigation)

## 6. Альтернативы (рассмотрены и отклонены, включая «delegated to agent»)

## 7. Открытые вопросы (resolved / deferred / research subtask)

## 8. DoD эпика (чеклист условий для /release)
```

**Источник истины:** plan = детальный контекст подзадач, backlog.md = state at a glance (компактные строки).

## 10. `/execute E-001` — детальный дизайн

### Концепция

«Автономно от старта до финала, диалог только в начале и в конце». Pre-execute confirmation (один раз) → автономный прогон всех волн → агрегированный отчёт + диалог.

### Команда

```
/execute E-001
```

ID эпика — единственный аргумент. **Без флагов.** Нет `--dry-run`, `--no-confirm`, `--resume-merges` и т.п. Доверяем агенту на этапе планирования и исполнения, аудит — в финальном диалоге.

### Orchestration mechanism

**Task tool subagents.** Главный агент (handler `/execute`) спаунит execution-agent'ов параллельно через несколько Task-вызовов в одном сообщении. Release-manager — тоже subagent.

- Subagents имеют независимый conversation context
- Файловая система общая, но каждый работает в своём worktree
- Иерархия: orchestrator → execution-agent (без вложенности; subagents не могут спаунить других)

### Context control (4 слоя)

1. **Стандартный** (auto, фиксированный): CLAUDE.md, путь к plan-E-001.md, спецификация подзадачи, путь worktree, имя ветки.
2. **Декларируемый** (control через `/plan`): поле «Контекст» в каждой подзадаче `plan-E-001.md` — список доп. файлов для чтения.
3. **Pre-execute summary** (один раз перед стартом): orchestrator печатает, что получит каждый subagent. Confirm `[y/a]`.
4. **Без `--dry-run`** — отказались, аудит только постфактум.

**Изоляция:** subagents не видят друг друга, conversation-history, чужие worktree.

### Execution-agent contract

#### Authority

| Может | Не может |
|---|---|
| Читать любые файлы в репо | Спаунить других subagent'ов |
| Писать / править в **своём** worktree | Пушить на remote (`git push`) |
| Запускать bash (git local, build, lint, test) | Модифицировать main или чужие worktree |
| `git worktree add`, commit на своей ветке | Спрашивать пользователя в реалтайме |
| Писать в `.task/report-E-001.X.md` | Использовать веб-инструменты |
| | Менять `.task/backlog.md`, `plan-E-001.md` |

#### Жизненный цикл

```
1. Receive prompt от orchestrator (Task tool)
2. git worktree add <path> -b feat/E-001.X feat/E-001
3. cd <worktree>
4. Read plan-E-001.md + свой subtask spec + декларированный контекст
5. Implement: edit / write / bash для тестов и сборки
6. Commit(s) на feat/E-001.X
7. Smoke check (по DoD)
8. Write .task/report-E-001.X.md
9. Return summary orchestrator'у
```

#### Возврат orchestrator'у (короткий summary)

```
Status: Done | Failed | Partial
Subtask: E-001.X
Branch: feat/E-001.X
Commits: 3
  - sha1, sha2, sha3
Smoke check: pass | fail | n/a
Note: <одна строка> (опционально)
```

Полный контекст — в файле отчёта. Возврат = карта для orchestrator'а.

#### Обстукло

**Когда:** файл из спеки не существует; smoke check fail после одной попытки; spec противоречит сама себе; развилка требует решения уровня пользователя; bash падает с непонятной ошибкой.

**Что делает:**
1. **Не делает rollback** — worktree и partial commits сохраняются для инспекции
2. Пишет секцию **Obstacles** в отчёт (tried / stopped at / need)
3. Возвращает **Status=Failed**
4. Cascade — на orchestrator'е (DAG из плана)

**Не пытается** обойти спеку, не зацикливается на retry, не возвращает success при фундаментальном фейле. Bash hang >10 минут → прерывает.

#### Лимиты

Один execution-agent = одна Task tool сессия = одна subtask.

Если context exhausted — Failed, obstacle = «context exhausted, recommend split». Превентивных эвристик **нет**, реактивный fallback в финальном диалоге.

### Smoke check

**Гибрид:** явная команда из DoD приоритетна, иначе type-based default.

| Тип | Default smoke check |
|---|---|
| `impl` | Тесты для затронутых файлов + lint + build |
| `refactor` | Существующие тесты зелёные |
| `test` | Тесты, которые написала эта подзадача, проходят против `feat/E-001` |
| `docs` | Markdown lint / link check, иначе файл существует |
| `research` | Research-файл существует и содержит 5 секций |

**Поведение при fail:**
- Одна попытка авто-фикса
- Не помогло → Status=Failed, obstacle с command/exit/output/tried/stopped at
- Не пытается отключать тесты ради «прохождения»

### Worktree management

#### Pre-execute analysis (когда есть стейт от прерванного прогона)

Anti-data-loss: orchestrator не чистит автоматически. Анализ → диалог → адаптация плана.

| Состояние | Default action |
|---|---|
| Worktree + commits + report Status=Done + smoke OK | Skip, mark Done в текущем плане |
| Worktree + commits + report Status=Partial | Resume / Restart |
| Worktree + commits + НЕТ отчёта | Resume / Restart |
| Worktree + commits + report Status=Failed | Restart / re-open /plan / Skip |
| Только ветка без worktree | Recreate worktree (resume) |
| Worktree без коммитов | Clean safe |
| **Uncommitted changes** | **Hard stop** — ручной разбор |

**Resumed подзадача** получает в prompt'е секцию «Resume context» с описанием уже сделанного.

#### Cleanup после Done

**Удаляет release-manager** (в merge mode) после успешного merge'а в `feat/E-001`. Failed → worktree сохраняется для инспекции. Branch удаляется аналогично.

#### Conflicts

**Не пытается резолвить автоматически. Анализирует и предлагает решение.**

release-manager на конфликте:
1. `git merge --abort` (откатывает попытку, не применяет)
2. Пишет в conflict-секцию report'а: что произошло, hunks, intent analysis, **proposal с confidence (high/medium/low)**, предложенный код
3. Worktree и ветка сохраняются
4. Продолжает с другими подзадачами волны

В финальном диалоге пользователь видит per-conflict:
- `[a]` Accept proposal — release-manager применяет резолв и доделывает merge
- `[r]` Reject, manual — выходим, пользователь резолвит руками
- `[s]` Show full diff
- `[k]` Skip merge для этой подзадачи

### Per-subtask report — `.task/report-E-001.X.md`

Шесть **обязательных** секций (даже если пустые `_none_`):

```markdown
# Report — E-001.X

**Status:** Done | Failed | Partial
**Subtask:** E-001.X — Title
**Type:** impl
**Started:** ISO timestamp
**Finished:** ISO timestamp
**Worktree:** .worktrees/E-001.X
**Branch:** feat/E-001.X
**Commits:** [list]

## Done
<bullet list, что реализовано + результат smoke check>

## Decisions
<автономные выборы: AD-X с Choice / Why / Re-visit if>

## Obstacles
<пусто для Done; для Failed/Partial — tried / stopped at / need>

## Open questions
<замечания, требующие решения уровнем выше>

## Follow-ups
<кандидаты на новые подзадачи: тип + краткое описание>
```

### Aggregated report — `.task/report-E-001.md`

Пишет release-manager в aggregation-режиме, читая все per-subtask отчёты + свои merge-findings + результат code-reviewer'а.

**Секции в порядке важности:**

1. Summary (status, stats)
2. Wave breakdown
3. **Obstacles** (failed subtasks с tried/stopped at/need)
4. **Conflicts** (с proposals от release-manager)
5. **Review** (от code-reviewer, severity Critical/High/Medium/Low)
6. Open questions (агрегат)
7. Decisions (агрегат)
8. Follow-ups suggested
9. Integration state (что в feat/E-001, kept worktrees/branches)

### Code-reviewer pass перед финальным диалогом

**Внутренний агент**, тот же `code-reviewer` что в плагине, но с release-readiness промптом. Запускается всегда (если эпик не failed целиком).

**Читает:** feat/E-001 после всех merge'ей (полный diff vs main), aggregated report, plan-E-001.md, CLAUDE.md.

**Ищет:**
- Cross-subtask issues (дублирование, дрифт стилей)
- Integration concerns (от сочетания merge'ей)
- Quality на epic-level (тесты, документация, naming)
- Соответствие плану

**Не делает:** не переписывает код, не запускает тесты, не блокирует.

**Findings** нумеруются `R-X`, попадают в секцию Review агрегированного отчёта.

### Inter-wave integration

```
Wave W:
  1. Spawn N execution-agent через Task tool параллельно
  2. Wait all returns
  3. Spawn release-manager (merge mode) для волны W
  4. Merge-loop: для каждой Done — git merge --no-ff в feat/E-001
     - Success: cleanup worktree + delete branch
     - Conflict: abort merge, conflict в report (proposal + confidence), worktree сохраняется
     - Failed/Partial: не мёрджим
  5. Optional post-wave smoke check на feat/E-001
  6. Cascade: orchestrator вычисляет downstream Skipped по DAG
  7. → Wave W+1
```

**Никакой интерактивности между волнами.** Один pre-execute confirm + автономия до финального диалога.

### Final dialog

#### Step 1 — narrative summary (структура + живой тон)

Скелет фиксированный, тон conversational:

```markdown
Epic E-001 — отчёт

## Сделано
<connected текст что закрыто, что в feat/E-001>

## Не сделано
<failed / skipped / blocked, конкретно>

## Что это значит
<implication для проекта: что не работает, чего не хватает>

## Требуются действия
<главная развилка с рекомендацией и обоснованием>

## Иначе
<последствия каждого пути: accept partial / defer>

## Дополнительно
<обзор decisions / open questions / follow-ups / review findings>

## Что обсуждаем?
<меню>
```

Длина — по содержанию, без искусственных лимитов.

#### Step 2 — финальное меню (фиксированный порядок по убыванию важности)

```
1. Obstacles         (X failed + Y skipped)
2. Review findings   (Critical / High / Medium / Low)
3. Open questions    (N)
4. Decisions         (M)
5. Follow-ups        (K)
6. Show full report
7. Skip, go to final action
```

Стрелка `← рекомендую начать здесь` ставится туда, где реально требует внимания (Obstacles → Review Crit/High → Open → ничего).

#### Step 3 — sub-dialogs по выбранным темам

Каждая категория имеет свои действия:

- **Obstacles:** Re-plan / Replace DoD and re-run / Skip permanently / Manual intervention
- **Review (Critical/High):** Fix now (mini-execution-agent) / Acknowledge as known limitation / Abort release
- **Review (Medium/Low):** Accept / Fix / Defer
- **Open questions:** Resolve now / Defer / Promote to research subtask
- **Decisions:** Accept / Mark for re-visit / Revert (новая subtask)
- **Conflicts:** Accept proposal / Reject (manual) / Show diff / Skip merge
- **Follow-ups:** Accept (→ backlog) / Reject

#### Step 4 — финальное действие

```
[r] /release E-001         — текущее состояние feat/E-001 готово к релизу
[i] Iterate                — re-plan failed/blocked, /plan E-001 phase 2
[p] Partial /execute       — перезапустить только failed/skipped без re-plan
[d] Defer                  — оставить эпик в In Progress, выйти
[a] Abort epic             — откатить feat/E-001, эпик → Cancelled
```

**Нельзя выйти без явного финального действия** — orchestrator переспрашивает.

## 11. `/release E-001` — детальный дизайн

### Концепция

**Агент готовит материалы, не релизит сам.** `/release` пишет файлы локально, коммитит на `feat/E-001`, печатает инструкции пользователю. **Не делает** `git merge → main`, `git tag`, `git push`. Это руки пользователя.

### Команда

```
/release E-001
```

ID эпика — обязательный. Без флагов.

### Высокоуровневый flow

```
1. Verify preconditions
   - Эпик в In Progress, feat/E-001 существует, есть aggregated report
2. Independent pre-release review
3. Determine epic state (Success / Partial / Failed / Conflicts)
4. Suggest version bump
5. Prepare release materials (drafts по одному)
6. Commit на feat/E-001 ("release: prepare vX.Y.Z")
7. Update backlog (E-001 → Done с датой)
8. Print user instructions (НЕ ВЫПОЛНЯЕТ)
```

### Чёткое разделение ответственности

| Что | Агент | Пользователь |
|---|---|---|
| Independent review feat/E-001 | ✅ | — |
| Update CHANGELOG / version files | ✅ | проверяет, корректирует |
| Commit на feat/E-001 | ✅ | — |
| Update backlog (Done) | ✅ | — |
| `git checkout main && merge` | — | ✅ |
| `git tag` | — | ✅ |
| `git push && git push --tags` | — | ✅ |
| GitHub Release | — | ✅ |

### Independent pre-release review

**Тот же `code-reviewer` agent**, но с release-readiness промптом — другой фокус, чем review в `/execute`.

| Проверяется | Не проверяется (это сделал review в /execute) |
|---|---|
| Public API / surface changes — задокументированы? | Cross-subtask code duplication |
| Breaking changes — пометка в CHANGELOG? | Integration concerns |
| Migration steps — достаточно? | Code style |
| README отражает новые команды? | |
| ARCHITECTURE.md / OVERVIEW.md обновлены? | |
| Hooks обновлены под новый формат? | |
| Версии в plugin.json / marketplace.json consistent? | |

**Findings:** `PR-X` (Pre-Release-1, ...). Severity: Critical / High / Medium / Low.

**Действия:**
- **Critical / High блокируют release** — Fix now (mini-execution-agent для тривиальных правок) / Acknowledge as known limitation (внести в CHANGELOG секцию Known Limitations) / Abort release
- **Medium / Low** — Accept / Fix / Defer

### Partial release policy

| Status в aggregated report | Поведение `/release` |
|---|---|
| `Success` | Прямой путь, без доп. подтверждений |
| `Partial` | Подтверждение «релизим как есть» + auto-секция Known Limitations в CHANGELOG из failed/skipped |
| `Failed` | Отказ работать (нечего релизить) |
| `Unresolved conflicts` | Отказ работать, пока конфликты не закрыты |

### Version bump heuristic

**Агент предлагает с обоснованием, пользователь подтверждает.**

| Изменения | Pre-1.0 (0.x.y) | 1.0+ |
|---|---|---|
| Только bug fixes / docs / refactor | patch | patch |
| Новые фичи / non-breaking | minor | minor |
| Breaking changes | minor | major |

В pre-1.0 любые breaking → minor по convention (Cargo / npm / etc).

При смешанных изменениях агент показывает все triggers и lean.

### Release artifacts (набор)

| Артефакт | Файл | Аудитория |
|---|---|---|
| **CHANGELOG entry** | `CHANGELOG.md` (append) | Технический, Keep a Changelog |
| **Release notes** | `.task/release-notes-vX.Y.Z.md` | Для GitHub Release — пользовательский язык, highlight'ы, install/upgrade |
| **Announcement** | `.task/announcement-vX.Y.Z.md` | Для TG / блога / social — короткий, цепляющий |
| **Tag message** | (текст, не файл) | 2–4 строки для `git tag -a` |
| **Migration guide** | `.task/migration-vX.Y.Z.md` (только при breaking) | Шаги обновления |

**Источники для генерации:** aggregated report, `git diff main..feat/E-001`, plan-E-001.md, existing CHANGELOG (для тона).

**Диалог по артефактам — пошаговый, не одним полотном.** По каждому: accept / edit / regenerate.

### Multi-epic release

**Структурно невозможно** из-за «один эпик In Progress за раз»: пока `/release E-001` не закроет E-001, нельзя начать E-002. Каждый эпик → один релиз.

### Rollback policy

**Точки abort'а внутри `/release`:**

| До шага 6 (commit) | Просто выход, ничего откатывать |
|---|---|
| После шага 6/7 | Нужны действия — `/release` НЕ делает auto-undo, печатает точные инструкции пользователю |
| После пользовательского merge на main / tag | За пределами `/release`, стандартный git |

**Edge case:** запуск `/release` на эпике с уже подготовленным release commit — pre-execute analysis детектит, предлагает Continue / Rollback / Abort.

### Release-manager modes (агент с тремя режимами)

| Режим | Где | Что делает |
|---|---|---|
| **merge** | `/execute` между волнами | Merge subtask-веток в `feat/E-001`, cleanup worktree'ев Done подзадач, conflict report с proposals |
| **aggregation** | `/execute` после всех волн | Собирает `.task/report-E-001.md` из per-subtask отчётов + merge findings + (после code-review pass) review findings |
| **full** | `/release` | Independent pre-release review (через code-reviewer), generate материалов, commit на feat/E-001, update backlog, print instructions |
